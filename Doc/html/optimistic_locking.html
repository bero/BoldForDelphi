Optimistic Locking

Handle concurrent access conflicts gracefully

Overview

Optimistic locking allows multiple users to work with the same data concurrently. Instead of locking records during editing, Bold detects conflicts when saving and lets you handle them appropriately.

This approach maximizes concurrency while ensuring data integrity when conflicts occur between users editing the same objects.

How It Works

Read Without Locks

Users can read and edit data without blocking others

Version Tracking

Each object tracks its version/timestamp from when it was loaded

Conflict Detection

On save, Bold checks if the database version matches what was loaded

Conflict Resolution

When conflicts occur, you decide: overwrite, reload, or merge

Testing Conflicts

  1. Run two instances of the application
  2. In both instances, load the same object
  3. Edit the object in Instance A and save
  4. Edit the same object in Instance B and try to save
  5. Instance B will detect the conflict - the data was modified since it was loaded
  6. Choose how to resolve: reload fresh data, force overwrite, or cancel

Implementation Notes

Bold uses a timestamp or version column to track object versions. When saving:

  • Bold compares the loaded version with the current database version
  • If they match, the save proceeds and the version is incremented
  • If they differ, a conflict is raised for you to handle

This is more efficient than pessimistic locking for read-heavy applications where conflicts are rare.
